Implementation plan — style-trans-fair

1. Task formulation

  • Task kind: multi-class classification (3 content classes per task) with group-fairness evaluation.
  • Input shape: tabular numeric vector of 20 features (toy substitute for Neural-Style-Transfer image features; same structure, CPU-feasible on HF Spaces ≤16 GB).
  • Output shape: one of 3 integer class labels {0, 1, 2} (one prediction per test row, written one per line in predictions.txt).
  • What is predicted: the content class of a stylised image. Why: to investigate whether models can generalise across style groups despite a heavily skewed training distribution — the central fairness question of the STYLE-TRANS-FAIR challenge. How scored: geometric mean of per-group accuracy across all 9 (content × style) groups (§3). Whether the data supports it: yes — the synthetic dataset faithfully replicates the 9-group imbalance structure from the paper (3 dominant groups × 20 samples, 6 minor groups × 5 samples in training; 10 per group in the balanced test set). What for: any high-stakes classifier trained on data that is biased by a spurious attribute (artistic style, background, demographic proxy).
  • Submission protocol: λ (result-submission). Participants train locally and upload predictions.txt — one predicted integer label per line, in the same row order as test_features.csv. Phase 2 should default to λ; γ (code-submission) doubles bundle complexity and is deferred to v2. Cite Pavão et al., Ch. 2 §2.1.

2. Data & splits

Generation (Phase 2 must implement this in the bundle without internet access)

  1. Call sklearn.datasets.make_classification(n_samples=270, n_features=20, n_informative=10, n_redundant=5, n_classes=3, n_clusters_per_class=2, random_state=42) → arrays X (270 × 20) and y (270,) of content labels.

  2. Assign style groups: iterate over the 270 samples in order.

  3. Allocate samples to the 9 (content, style) groups using the following target counts:
    • Dominant groups — (content=0, style=0), (content=1, style=1), (content=2, style=2): 20 training + 10 test samples each.
    • Minor groups — the remaining 6 (content, style) pairs: 5 training + 10 test samples each.
  4. Concretely: for each content class c ∈ {0,1,2}, take the make_classification samples with label c (≈90 samples each), sort them by index, and assign:
    • first 20 → train, style=c (dominant)
    • next 5 → train, style=(c+1)%3 (minor)
    • next 5 → train, style=(c+2)%3 (minor)
    • next 10 → test, style=c
    • next 10 → test, style=(c+1)%3
    • next 10 → test, style=(c+2)%3 This gives exactly 90 train rows and 90 test rows.

  5. Encode group as string "<content>_<style>", e.g. "0_0", "1_2".

  6. Inject style bias into features: for each sample with style group s, add an offset to the first 3 features only:

  7. style 0 → [+2.0, 0.0, 0.0]
  8. style 1 → [0.0, +2.0, 0.0]
  9. style 2 → [0.0, 0.0, +2.0]
  10. Features 3–19 are unchanged. This makes naïve models learn style-correlated shortcuts instead of true content signal.

3. Metric

The scoring program computes per-group accuracy for each of the 9 groups using sklearn.metrics.accuracy_score(y_true[mask_g], y_pred[mask_g]) where mask_g = (test_df["group"] == g). Let accs be the list of 9 per-group accuracies.

  • Primary — Geometric Mean Accuracy (GM): scipy.stats.gmean(numpy.clip(accs, 1e-9, None)) Direction: higher is better. Written to scores.json as key "geometric_mean_accuracy". Rationale: GM collapses to ≈0 if any group has near-zero accuracy, directly incentivising participants to fix minority-group performance rather than over-fitting to dominant groups.

  • Secondary 1 — Worst Group Accuracy (WGA): float(numpy.min(accs)) Written to scores.json as key "worst_group_accuracy". Higher is better.

  • Secondary 2 — Arithmetic Mean Accuracy (AM): float(numpy.mean(accs)) Written to scores.json as key "arithmetic_mean_accuracy". Higher is better.

  • Confidence interval: bootstrap N=100 on the GM score (resample test rows with replacement, recompute GM, report 5th–95th percentile). Written to scores.json as "gm_ci_low" and "gm_ci_high".

  • Leaderboard primary sort key: geometric_mean_accuracy descending.

Cite Pavão et al., Ch. 4 for metric-choice rationale.

4. Baseline

Trivial baseline — sanity floor

sklearn.dummy.DummyClassifier(strategy="most_frequent", random_state=42)

Trained on train.csv. Predicts the most frequent training label for every test row. Expected scores: AM ≈ 0.33 (one-class majority), GM ≈ 0.0 (predicts 0 for 6 of 9 groups), WGA = 0.0. Exported as solution/sample_code_submission/dummy_predictions.txt (the starting-kit "zero-effort" example submission).

Modest baseline — standard logistic regression (no fairness correction)

sklearn.linear_model.LogisticRegression(max_iter=1000, random_state=42)

Trained on features f0…f19 from train.csv, labels from label column. Expected scores on the balanced test set: AM ≈ 0.55–0.65, GM ≈ 0.30–0.45, WGA ≈ 0.10–0.25 (depressed by the 3 groups under-represented in training). This model is exported as solution/sample_code_submission/model.py in the bundle.

Fair-hint baseline (mention in the overview page, not a separate executable)

sklearn.linear_model.LogisticRegression(class_weight="balanced", max_iter=1000, random_state=42) — uses inverse-frequency class weights to partially compensate for the training imbalance. Expected improvement: GM ≈ 0.45–0.55, WGA ≈ 0.25–0.35. Mention in the competition overview page as a concrete starting point.

5. Rules

  • Submission caps: Development phase — 5 submissions per day per team. Final phase — 1 submission per team (single definitive model).
  • Anti-cheating posture:
  • Multi-account detection: yes (enable Codabench built-in flag).
  • Test-label leakage: test_labels.csv is kept exclusively in reference_data/ (never shipped in the participant download).
  • Ensemble-of-submissions fraud detection: no (deferred to v2).
  • Paraphrase / re-prompt defense: not relevant (tabular classification task).
  • Winner-code release: required — winners must submit a reproducible code package within 2 weeks of the final leaderboard freeze.
  • Reproducibility check: yes — organiser manually re-runs the winner's code on an additional held-out balanced set of 18 test samples (2 per group) generated with random_state=99 (same pipeline, different seed) to verify the reported GM score.
  • Cite Pavão et al., Ch. 2 §2.4.

6. Ethics & dual-use

  • Dual-use risk: Low. A strong bias-mitigation solution benefits practitioners building fairer classifiers in medicine, hiring, and content moderation. No harmful application identified for style-invariant image classification.
  • Privacy: Fully synthetic data — no real individuals, no identifying features, no consent required.
  • Fairness: The challenge is a fairness benchmark; the three metrics explicitly surface majority-vs-minority group performance. The protected attribute is "artistic style" (a controlled spurious correlation), not a demographic attribute — so no direct demographic harm, but the lesson generalises to demographic proxies.
  • Datasheet: deferred — a full datasheet is needed when the real Meta-Album + Neural Style Transfer dataset is used for the v2 challenge.
  • IRB / consent: not applicable (synthetic data, no human subjects).
