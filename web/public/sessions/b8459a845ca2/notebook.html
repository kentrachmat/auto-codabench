Section 1 — Task Formulation¶

5W: What, Why, How, Whether, What For¶

W Answer
What Predict the content class (one of 3 image categories) from a neural-style-transferred image
Why Expose and measure style-induced bias: models trained on imbalanced content×style data tend to conflate style with content
How Scored by Geometric Mean Accuracy across all 9 (content, style) groups — penalises collapse onto dominant groups far more than accuracy does
Whether Meta-Album provides richly labelled multi-domain images; Neural Style Transfer (Gatys et al. 2015) introduces controlled, reproducible bias at scale
What For Benchmarking bias-invariant vision models; fairness auditing toolkits for deployed classifiers in medical imaging, ecology, and document understanding

Submission protocol: γ (code submission) — participants submit a model.py file; the platform executes it on held-out test data within a 3 600 s wall-clock budget. This prevents label leakage and enforces reproducibility (Pavão et al., Ch. 2 §2.4).

Per Pavão et al., Ch. 2 §2.1, a well-posed task must specify the split unit, prediction target, and evaluation protocol before data is shown to participants.

Section 2 — Data & Splits¶

Source & License¶

  • Content images: Meta-Album Set 2 — a multi-domain meta-dataset covering animals, plants, microscopy, vehicles, human actions, and more. 40 datasets across 10 domains; publicly available for academic use (Ullah et al. 2022).
  • Style images: 8 000 images scraped from Google Images and filtered by an SVM quality classifier (88% test accuracy) to retain 80 "good" style classes. Used under a research / educational fair-use posture; a Datasheet for the Dataset will be authored before the competition goes live.
  • Stylised images: generated by Neural Style Transfer (Gatys et al. 2015) with VGG-19 at 256×256 px.

Split policy (Pavão et al., Ch. 3 §3.2)¶

Each task (the atomic evaluation unit) contains exactly:

Split Images Groups Distribution
Biased train 90 9 (3 content × 3 style) Diagonal groups 4× oversampled (20 imgs vs 5 imgs)
Balanced test 180 9 20 images per group

~200 tasks in the development phase; ~200 tasks in the final phase.

Toy demo (this notebook): synthetic stand-in with identical 3×3 structure. Replace read_data() with your real loader after downloading from the competition dataset repo.

import numpy as np
import pandas as pd
import matplotlib.pyplot as plt
from collections import Counter

def read_data(seed=42):
    """Toy stand-in for STYLE-TRANS-FAIR (one task).

    Mirrors the canonical 3×3 group structure from §2.3 of the paper:
      - Biased train  : 90 images  (diagonal groups 4× oversampled)
      - Balanced test : 180 images (20 per group)

    Real usage: replace this body with a loader that reads
      `stylized/` images + `labels.csv` from the competition dataset.
    Feature vector = flattened pixel values; here N_FEAT=32 for speed.
    """
    rng = np.random.RandomState(seed)
    N_FEAT = 32   # toy dim — real: 256*256*3 = 196 608 after flatten/pool

    # Biased training distribution (paper Fig. 10 / §2.3)
    train_counts = {
        (0, 0): 20, (0, 1): 5,  (0, 2): 5,
        (1, 0): 5,  (1, 1): 20, (1, 2): 5,
        (2, 0): 5,  (2, 1): 5,  (2, 2): 20,
    }
    test_counts = {(c, s): 20 for c in range(3) for s in range(3)}

    def _make_group(content, style, n):
        # Content: a distinct mean in a dedicated feature sub-range
        mu = np.zeros(N_FEAT)
        mu[content * (N_FEAT // 3):(content + 1) * (N_FEAT // 3)] = 2.0
        # Style: controls noise magnitude (mimics texture-strength variation)
        sigma = 0.5 + style * 0.5
        return rng.randn(n, N_FEAT) * sigma + mu

    def _build_split(counts):
        Xs, ys, gs = [], [], []
        for (c, s), n in counts.items():
            Xs.append(_make_group(c, s, n))
            ys.extend([c] * n)
            gs.extend([(c, s)] * n)
        return np.vstack(Xs), np.array(ys), gs

    tr_X, tr_y, tr_g = _build_split(train_counts)
    te_X, te_y, te_g = _build_split(test_counts)
    return dict(train_X=tr_X, train_y=tr_y, train_groups=tr_g,
                test_X=te_X,  test_y=te_y,  test_groups=te_g)

Section 3 — Metric¶

Primary: Geometric Mean Accuracy (GM)¶

For a task with 9 groups indexed $i = 1, \ldots, 9$, let $a_i$ be the per-group classification accuracy. The three metrics are:

$$\text{GM} = \left(\prod_{i=1}^{9} a_i\right)^{1/9}, \qquad \text{AM} = \frac{1}{9}\sum_{i=1}^{9} a_i, \qquad \text{WGA} = \min_i(a_i)$$

Why GM as the primary metric? A model that ignores minority style groups and achieves 0 % on even one group gets GM ≈ 0 — the geometric mean propagates failure multiplicatively. This is stricter than AM (which lets dominant-group wins mask minority failures) but less brittle than WGA (which is a single worst-case sample). The competition chose GM as the metric that best incentivises balanced performance across all nine groups (Pavão et al., Ch. 4).

AM and WGA are displayed on the leaderboard as diagnostic signals so participants can understand where their model underperforms.

def compute_metrics(y_true, y_pred, groups):
    """Compute AM, GM, and WGA across all (content, style) groups.

    Args:
        y_true  : 1-D array of true content labels.
        y_pred  : 1-D array of predicted content labels.
        groups  : list of (content, style) tuples, one per sample.

    Returns:
        dict with keys 'gm', 'am', 'wga', 'per_group'.
    """
    y_true = np.asarray(y_true)
    y_pred = np.asarray(y_pred)
    unique_groups = sorted(set(groups))
    accs = []
    for g in unique_groups:
        mask = np.array([gi == g for gi in groups])
        acc = float(np.mean(y_pred[mask] == y_true[mask]))
        accs.append(acc)
    accs = np.array(accs)
    gm  = float(np.exp(np.mean(np.log(np.clip(accs, 1e-9, None)))))  # geometric mean
    am  = float(np.mean(accs))
    wga = float(np.min(accs))
    return {
        "gm":        gm,
        "am":        am,
        "wga":       wga,
        "per_group": {str(g): round(float(a), 4)
                      for g, a in zip(unique_groups, accs)},
    }

def score(y_true, y_pred, groups):
    """Return the primary metric (GM) — convenience wrapper for the scoring program."""
    return compute_metrics(y_true, y_pred, groups)["gm"]

Section 4 — Baseline & Starting Kit¶

This section is the participant-facing starter code. It provides two baselines:

Baseline Strategy Expected GM
Trivial Majority-class predictor ≈ 0 (collapses minority groups)
Modest Logistic Regression on raw features > chance; limited by linear separability

The paper's best result used EfficientNetv2 + group resampling (GM ≈ 0.65 on real data, §5.1.3). On toy data here the modest baseline already scores well because content signal is easy — the gap on real stylised images is the challenge.

Both baselines implement .fit() / .predict() so the code can be dropped directly into sample_code_submission/model.py for a γ submission.

# ── Trivial baseline: majority-class predictor ────────────────────────────────
class TrivialBaseline:
    """Ignores all features; returns the most frequent class seen in training.

    Sets the sanity floor: any real model should beat this.
    On a biased dataset, this will score well on AM but near-0 on WGA/GM.
    """

    def fit(self, X, y):
        classes, counts = np.unique(y, return_counts=True)
        self.majority_class_ = int(classes[np.argmax(counts)])
        return self

    def predict(self, X):
        return np.full(len(X), self.majority_class_, dtype=int)

# ── Modest baseline: Logistic Regression ─────────────────────────────────────
from sklearn.linear_model import LogisticRegression
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import Pipeline

class ModestBaseline:
    """Logistic Regression on standardised raw features.

    Fast, interpretable, and a good starting point for participants.
    In the real competition, replace the feature extractor with a
    pre-trained CNN backbone (e.g. EfficientNetv2, ResNet-50).

    The `.fit()` / `.predict()` interface is the contract expected by
    the competition's ingestion program.
    """

    def __init__(self, C=1.0, max_iter=500):
        self.pipeline = Pipeline([
            ("scaler", StandardScaler()),
            ("clf",    LogisticRegression(C=C, max_iter=max_iter,
                                          random_state=42, n_jobs=-1)),
        ])

    def fit(self, X, y):
        self.pipeline.fit(X, y)
        return self

    def predict(self, X):
        return self.pipeline.predict(X).astype(int)

Section 5 — Rules¶

Submission protocol¶

  • γ (code submission): participants submit a model.py file containing a class with .fit() / .predict() methods. The Codabench platform executes it on the held-out test set within a 3 600 s wall-clock budget on an 8-core x86-64 CPU worker with up to 56 GB RAM (Pavão et al., Ch. 2 §2.4).

Submission caps¶

Phase Submissions Leaderboard
Development 5 per day Public — visible to all
Final 1 total Hidden until phase closes

Participants may revise their selected final submission at any time before the final phase closes, but the last submitted entry counts.

Anti-cheating¶

  • Multi-account submissions are prohibited; Codabench account verification is required at registration.
  • Participants may not use the held-out test labels in any form. The solution file hash is logged by the platform before the competition opens.
  • Ensemble fraud (averaging multiple independently trained models) is permitted, but the submitted model.py must be self-contained and produce the leaderboard score when re-run by the organisers.
  • Participants may not share test-set predictions with each other during the competition.

Winner-code release (required)¶

The top-3 finalists must release their complete training code (including model.py + training script + requirements.txt) under an open-source licence within 30 days of the final leaderboard publication. Submissions whose score cannot be reproduced within ±0.5 % GM will be disqualified.

Reproducibility check¶

Organisers will re-run the winner's submitted model.py on the final test set and verify the score matches the leaderboard within ±0.5 %. The re-run environment uses the same Codabench worker specifications (8-core CPU, 56 GB RAM, 3 600 s budget).

Section 6 — Ethics & Dual-Use¶

Dual-use risk — Low¶

The primary beneficiaries of a strong solution are fairness-aware ML practitioners: researchers building bias-resistant vision classifiers for medical imaging (e.g. dermatology, pathology), ecological monitoring, and document analysis. The competition does not involve surveillance, biometric identification, generative deception, or weapons-adjacent technology. No dual-use concern has been identified.

Privacy¶

  • Meta-Album content images contain animals, plants, microscopy slides, remote-sensing tiles, textures, and vehicles. No personally identifiable information (PII) is present; no human faces or biometric data appear in the dataset.
  • Scraped style images were collected from Google Images. While some style images may incidentally contain recognisable artworks or trademarked designs, they are used exclusively as neural-style textures and are never the prediction target. The collection method is documented in §3.2 of the companion paper.
  • A Datasheet for the Dataset (Gebru et al. 2021) will be authored before the competition goes live, documenting: collection methodology, filtering pipeline, licence status, and any identified content concerns.

Fairness¶

The competition is itself a fairness benchmark — the explicit design goal is to reduce worst-group accuracy gaps induced by style bias. No demographic or linguistic attributes of participants are collected or used for ranking. Organisers will actively recruit participants from underrepresented research communities (Latin America, Africa, Southeast Asia) via partner institutions.

IRB / Consent¶

No human-subjects data are collected or used in the competition dataset. IRB review is not required. Participants submit only executable code; no personal data beyond standard Codabench account information (governed by the Codabench platform privacy policy) is retained by the organisers.

Known limitation¶

The style-transfer pipeline was validated on 20 random trials per style class (§3.2.3), but style fidelity was not audited across all 10 content domains. Domain-specific failures (e.g. microscopy images where style transfer destroys diagnostic texture) should be surfaced in the Datasheet before a large-scale deployment of this benchmark.
